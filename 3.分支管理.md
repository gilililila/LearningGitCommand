当我们在GitHub中与他人合作完成一个项目的时候，我们不会直接修改主分支，我们所有的修改将在其他分支中进行，确认无误后才合并到主分支中。

- **Master分支** 是已经确定的分支。
- **Feature分支** 相当于草稿纸，可以随意修改，不会影响到主分支。

### 查看分支
首先确定自己的分支在哪。在终端处输入
```Bash
git branch
```
我们应该会看到`* master`，星号表示当前所在的位置。

### 创建并切换到“草稿”分支
假设我们现在需要给笔记仓库添加一个`README.md`的介绍文件。我们需要创建一个`feature/add-readme`分支。
使用命令：
```Bash
git switch -c feature/add-readme
```
- `-c`意为创建
- `feature/add-readme`是社区通用的命名规范，表示这是一个开发新功能的分支。
现在再次输入`git branch`，星号应该在`feature/add-readme`前。

### 在“草稿”分支中操作
1. 创建一个`README.md`文件。随便写点东西。
2. 按照常规操作提交更改。

当我们切换会主分支后，在“草稿”分支中提交的所有内容都会消失。
Git将电脑上的文件替换为`master`分支最后一次提交时的模样。

当我们再次回到“草稿”分支时，在“草稿”分支上提交的内容重新出现了。

### 分支合并
当我们在“草稿”分支上已经把内容写的很完美的，就可以把这些修改**合并**到主分支`master`上。
1. **回到主分支**
把东西合并到哪里，就先切换到哪里。我们是将新文档合并到`master`分支上，就先切换到`master`分支上。
```Bash
git switch master
```
2. **执行合并命令**
```Bash
git merge feature/add-readme
```

终端会出现类似`Fast-forward`的字样，并列出`README.md`和其他修改过的笔记文件的名字。现在，`master`分支上就会拥有“草稿”分支上修改的内容了。
